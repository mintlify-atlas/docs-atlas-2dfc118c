Dexer is a .NET library that gives you full read-write access to Android Dalvik Executable (`.DEX`) files — the binary format that contains compiled bytecode for the Android Runtime. Whether you are building analysis tools, security research utilities, or automated patching pipelines, Dexer lets you treat a DEX file as a structured object model rather than raw bytes.

## What you can do

Dexer exposes the complete structure of a DEX file through strongly-typed C# objects. Once you load a file, you can traverse every class, method, field, and annotation it contains, read or rewrite individual bytecode instructions, and serialize the result back to a valid DEX file that Android tooling will accept.

## Object model overview

At the top of the hierarchy sits the `Dex` class. It holds a flat list of `ClassDefinition` objects. Each `ClassDefinition` contains lists of `MethodDefinition` and `FieldDefinition` objects, along with access flags, annotations, implemented interfaces, and inner classes. Each `MethodDefinition` optionally carries a `MethodBody` that contains the list of `Instruction` objects you can read and rewrite.

```
Dex
└── ClassDefinition[]          (dex.Classes)
    ├── FieldDefinition[]      (cdef.Fields)
    └── MethodDefinition[]     (cdef.Methods)
        └── MethodBody
            ├── Instruction[]  (body.Instructions)
            └── Register[]     (body.Registers)
```

## Typical use cases

<AccordionGroup>
  <Accordion title="Android reverse engineering">
    Read a DEX file extracted from an APK, navigate to a specific class or method, and inspect its bytecode to understand what the application is doing at runtime.
  </Accordion>
  <Accordion title="Automated patching">
    Load a DEX file, locate a target method by class and method name, replace one or more instructions, and write the patched file back — without disassembling or recompiling the full APK.
  </Accordion>
  <Accordion title="Static analysis tooling">
    Iterate over every `ClassDefinition` and `MethodDefinition` in a DEX file to build call graphs, detect API usage patterns, or flag suspicious opcode sequences.
  </Accordion>
  <Accordion title="Test instrumentation">
    Inject method calls or logging instructions into existing methods before running tests, without modifying the original source code.
  </Accordion>
</AccordionGroup>
